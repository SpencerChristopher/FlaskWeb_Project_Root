feat: Implement Redis-backed rate limiting and secure JWT handling

This commit introduces a robust, Redis-backed rate limiting system and significantly hardens the frontend's JWT handling for improved security and reliability.

Rate Limiting:
- Migrates the rate-limiting backend from in-memory to Redis, ensuring consistent state across multiple processes and enabling effective IP-based limiting.
- Adds a Redis service to docker-compose.yml with memory constraints for safe execution on resource-constrained devices.
- Updates the Flask-Limiter configuration to use the new Redis storage URI.
- Enables and fixes the previously skipped rate-limiting test suite, which now validates the Redis implementation.

Security & Frontend:
- Refactors the frontend JavaScript to remove JWTs from localStorage, mitigating XSS risks by relying solely on secure HttpOnly cookies for session management.
- Hardens JWT payload parsing on the frontend to prevent errors from malformed claims.
- Improves the login form's error handling to provide specific user-feedback for 429 (Too Many Requests) errors.

CI/CD & Testing:
- Adds a test execution step to the `test-deploy.yml` workflow, ensuring pytest is run automatically for all pushes to the dev branch.
- Fixes a critical bug in the `deploy.sh` script where an incorrect MONGO_URI was being used.
- Makes the deployment script more robust by checking for all required environment variables.
- Cleans up the test suite by removing old, redundant rate-limiting tests from the security hardening tests.
